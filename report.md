# Отчет по практической работе №1

## 1. UML-диаграммы

### 1.1. Диаграмма вариантов использования

![Диаграмма вариантов использования](use_case_diagram.puml)

**Описание:**

*   **Акторы:**
    *   **Соискатель гранта:** Подает, просматривает статус и отзывает заявки.
    *   **Независимый эксперт:** Оценивает назначенные ему заявки.
    *   **Держатель фонда:** Управляет пользователями, назначает экспертов, просматривает результаты экспертизы и принимает окончательное решение по заявкам.

*   **Варианты использования:**
    *   Общие для всех: `Авторизоваться в системе`.
    *   Для соискателя: `Подать заявку`, `Просмотреть статус заявки`, `Отозвать заявку`.
    *   Для эксперта: `Оценить заявку`, `Просмотреть назначенные заявки`.
    *   Для держателя фонда: `Принять решение по заявке`, `Назначить экспертов`, `Просмотреть результаты экспертизы`, `Управлять пользователями`.
    *   Все основные действия требуют авторизации (`<<include>>`).

### 1.2. Диаграмма классов

![Диаграмма классов](class_diagram.puml)

**Описание:**

*   **Пакет "Пользователи":**
    *   `User`: Абстрактный класс, определяющий общие атрибуты и методы для всех пользователей системы (ID, имя пользователя, email и т.д.).
    *   `Applicant`, `Expert`, `FundHolder`: Конкретные классы пользователей, наследующиеся от `User` и реализующие специфичные для их ролей функции.

*   **Пакет "Заявки":**
    *   `Application`: Основной класс, представляющий заявку на грант. Содержит информацию о заявке и ее текущий статус.
    *   `ApplicationStatus`: Перечисление (enum) возможных статусов заявки.
    *   `Evaluation`: Представляет оценку, данную экспертом конкретной заявке. Включает балл и комментарии.
    *   `GrantDecision`: Представляет окончательное решение по заявке, принятое держателем фонда.

*   **Отношения:**
    *   **Наследование (Generalization):** `Applicant`, `Expert` и `FundHolder` являются наследниками `User`.
    *   **Агрегация (Aggregation):** Соискатель (`Applicant`) "подает" заявки (`Application`). Заявки могут существовать в системе даже после удаления аккаунта соискателя.
    *   **Композиция (Composition):** Заявка (`Application`) "содержит" оценки (`Evaluation`) и решение (`GrantDecision`). При удалении заявки эти связанные объекты также удаляются.
    *   **Зависимость (Dependency):** Показаны логические зависимости, например, `FundHolder` зависит от `Expert` и `Application` при назначении экспертизы.

## 2. Ответы на контрольные вопросы

**1. Что такое UML? Какие вы знаете основные диаграммы UML?**

**UML (Unified Modeling Language)** — это унифицированный язык моделирования, который используется в объектно-ориентированном анализе и проектировании для визуализации, спецификации, конструирования и документирования программных систем.

Основные диаграммы UML делятся на две большие группы:

*   **Структурные диаграммы (Structure Diagrams):**
    *   **Диаграмма классов (Class Diagram):** Показывает структуру системы, классы, их атрибуты, методы и отношения между ними.
    *   **Диаграмма компонентов (Component Diagram):** Моделирует компоненты, из которых состоит система.
    *   **Диаграмма объектов (Object Diagram):** Показывает экземпляры классов и их связи в определенный момент времени.
    *   **Диаграмма развертывания (Deployment Diagram):** Описывает физическое размещение программных компонентов на узлах системы.
    *   **Диаграмма пакетов (Package Diagram):** Группирует элементы модели в пакеты.
    *   **Диаграмма композитной структуры (Composite Structure Diagram):** Показывает внутреннюю структуру класса.

*   **Поведенческие диаграммы (Behavior Diagrams):**
    *   **Диаграмма вариантов использования (Use Case Diagram):** Описывает функциональность системы с точки зрения внешних пользователей (акторов).
    *   **Диаграмма последовательности (Sequence Diagram):** Показывает взаимодействие объектов во времени, упорядоченное по времени.
    *   **Диаграмма деятельности (Activity Diagram):** Моделирует бизнес-процессы и потоки работ.
    *   **Диаграмма состояний (State Machine Diagram):** Описывает жизненный цикл объекта и его переходы между состояниями.
    *   **Диаграмма коммуникации (Communication Diagram):** Аналогична диаграмме последовательности, но акцентирует внимание на связях между объектами, а не на времени.
    *   **Диаграмма обзора взаимодействий (Interaction Overview Diagram):** Комбинация диаграммы деятельности и диаграмм последовательности.

**2. Какие элементы входят в состав диаграммы классов?**

В состав диаграммы классов входят следующие основные элементы:

*   **Классы (Classes):** Прямоугольники, представляющие собой шаблон для создания объектов. Класс имеет три секции:
    *   **Имя класса:** Название класса.
    *   **Атрибуты (Attributes):** Характеристики или свойства класса.
    *   **Методы (Operations/Methods):** Действия, которые может выполнять класс.

*   **Интерфейсы (Interfaces):** Определяют набор методов, которые должен реализовать класс, не предоставляя самой реализации.

*   **Отношения (Relationships):** Линии, соединяющие классы и интерфейсы:
    *   **Ассоциация (Association):** Отношение между двумя или более классами. Может быть направленной и иметь кратность (например, 1, 0..*, 1..*).
    *   **Агрегация (Aggregation):** Частный случай ассоциации ("часть-целое"), где "часть" может существовать независимо от "целого". Обозначается пустым ромбом.
    *   **Композиция (Composition):** Более строгая форма агрегации, где "часть" не может существовать без "целого". Обозначается закрашенным ромбом.
    *   **Наследование/Обобщение (Generalization):** Отношение "является" (is-a), где один класс (дочерний) наследует свойства и методы другого (родительского). Обозначается стрелкой с пустым треугольником.
    *   **Реализация (Realization):** Отношение, при котором класс реализует интерфейс. Обозначается пунктирной линией со стрелкой с пустым треугольником.
    *   **Зависимость (Dependency):** Отношение, при котором изменение в одном элементе (независимом) может повлиять на другой (зависимый). Обозначается пунктирной стрелкой.
